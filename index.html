Question: Your boss wants to know the top 10 video games of all time or top 10 for a specific year.

Question: Your boss wants to understand which genre is most popular. We'd like to see genre sales broken out per region.

Question: Lastly, your boss wants to know which publisher to pick based on which genre a game is.

Describe how your dashboard answers the questions presented. You don't have to address every question directly, but should at a high level address the main questions. (10 points)

Overall, my graphs tackle each question very simply. For the first question, I allow the user to select a specific year, and dynamically load the top games for that year. Checking that chart answers the first question. For the second question, I allow the user to select a region, and then a piechart is generated; this piechart very simply demonstrates the share of each genre for each region. You can also hover over each slice to get more information. For the third question, I allow the user to select a genre, and then a bar graph is generated representing the top publishers for that genre across all entries. As such, I believe that my boss would be quite happy -- all the info is there, and it's easy to understand! Also!!! I dynamically adjust the graphs based on screensize (in particular, check out how it looks on an iPhone :)) so it'll work no matter the screen!!!

List 3 reasons why D3 was helpful and improved your visualization (6 points)

  1. Speed of implementation. Instead of reinventing the wheel, you can just reuse the existing tools for making a wheel. As such, for each of the bar chart and pie chart implementations, I simply looked at documentation for D3, and was able to easily make a version of each that satisfied my needs. This improved visualization because we can use the best version of existing tools easily.
  2. Animations. D3 has nifty tools (namely .transition() and .duration()) for animation between versions of graphs. This was particularly useful when dynamically adjusting between options in each graph; for instance, change the region for the piechart, and you'll find that the piechart animates the transition between datasets!
  3. Flexibility. We clearly had different usecases, and different data input for each of the questions above. However, D3 allowed us to use the same framework to answer very different questions. There was no strict format that our data HAD to follow, and using different data was fairly easy. This allowed for the questions to be answered easily and more specifically!

List 3 reasons why D3 would not be the best tool for creating a visualization (6 points)

  1. Javascript itself isn't the right choice. If a tool needs to be written in some non-Javascript language, then using D3 becomes quite difficult. There are indeed ways to use D3 in Python, for example, but it becomes a bit more of a hassle. As such, I think if not using Javascript, I'd find a different library.
  2. The company you work for doesn't use D3. Yes, it would be great if you could choose the exact library and framework to use for everything you work on ever. But the reality is that if you work at a big company, odds are there will be standards and decisions made before you ever get there. So, if they decided to not use D3 (or tried data viz before D3 existed), then you're out of luck -- it's not the best tool, because there's no way it is reasonable to reimplement all of your company's data visualizations just to have D3.
  3. Javascript is disabled on the site. If you know that Javascript is going to be disabled on your site (for whatever reason, perhaps security reasons, who knows), then it would be impossible to use D3. Or, if you can't load external CDNs, then you can't load the library into memory. Either way, basically, if there's a scenario where restrictions are placed on your vanilla html/css project, then this would NOT be a time to add a depedency like D3.

Evaluate the accessibility of your dashboard based on the readings (1, 2) in the “Before you begin” section. What kinds of users might find this dashboard accessible and who might have more difficulty? What additional actions might you take to make this dashboard more accessible to all audiences? Your response should refer to at least one of the readings and be about 1 paragraph.

This reading was absolutely fascinating for better understanding our data readers. In particular, the first article says "It just so happens that this visual mapping doesn’t work for people who can’t see" in reference to a line graph. This made me realize that users who are blind may have immense difficulty using my dashboard. A solution to this would be to include aria-labels so that each data entry in each of the graphs can be screen read, for example. A different solution would be to "make those trends and relationships explicit to people who need it" -- for example, we don't need to screen read EVERY thing (unless they want that), instead we can also provide options like just asking for the max, or min, or highlighting trends, etc. My graphs are very accessible for users without any disabilities, including those that are colorblind (as I tried to choose colors that weren't difficult to parse for individuals with colorblindness). Overall, adding screen reading options and labels for HTML elements, as well as more descriptions that explain the trends / data without needing to see it.

Reflect on the stages of your design and implementation process when you could have taken steps to make your dashboard more accessible to all audiences. What are some factors that kept you from taking these steps? (a few sentences)

Naturally, and I think this will be the case for most other students, the biggest factor preventing me from implementing accessibility is that (1) I wasn't graded on it, and (2) I didn't have enough time. Unfortunately, with academic assignments, unless something is explicitly asked for, there's virtually no reason to do it; therefore, accessibility options sadly came second to completing the assignment. In a real world, customer facing scenario, this is also all too common -- if you have a deadline from your boss, and you need to finish your code, you're probably going to prioritize meeting the deadline over adding accessibility options (unless you're at Amazon, in which we'll just move the release back a bit to make sure the product is as customer-centric as we can get it). During the implementation process, when I was building each graph (see buildGraph1(), buildGraph2(), buildGraph3()), these would be good opportunities to make sure that I've thought extensively about the dashboard's accessibility. In particular, when deciding on colors, it would be a perfect time to think about individuals with color blindness; when thinking about labels, it would be important to think about blind users who need to use screen readers. Each feature has a different set of issues that comes along with it, so evaluating as we tackle each one could be a good approach. :)
